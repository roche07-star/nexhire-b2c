1. 배경 및 목적

🎯 핵심 질문

"PRO 헤드헌터의 경우 대시보드 + 파이프라인까지 제공하는게 좋지 않을까?"

- CTO님 제안 (2026.06.13)

배경

  • ▪ Jobizic Adam은 B2C (일반 구직자) + 개별 헤드헌터를 타겟으로 함
  • ▪ 현재 FREE 플랜은 분석만 가능 (대시보드/파이프라인 없음)
  • ▪ PRO 플랜 (₩19,900)은 개별 헤드헌터가 주 타겟이나, 후보자 관리 기능이 부족
  • ▪ EXPERT 플랜 (₩49,900)은 써치펌(B2B)을 위한 Eve 연동 + 고급 기능 제공

목적

  • ✅ PRO 플랜 가치 향상 - 개별 헤드헌터에게 필수적인 후보자 관리 기능 제공
  • ✅ 시장 경쟁력 강화 - 타 ATS 대비 압도적 가성비 (₩19,900 vs $300~800)
  • ✅ 명확한 플랜 차별화 - FREE (트라이얼) → PRO (개인용) → EXPERT (팀용)
  • ✅ EXPERT 업그레이드 동기 부여 - 간단한 파이프라인 사용 후 고급 기능 필요성 인지

4. 기술 스펙

4.1 DB 스키마 설계

기존 테이블 활용

-- analyses 테이블 (기존)
ALTER TABLE analyses
ADD COLUMN pipeline_stage TEXT DEFAULT 'pending';
-- 값: 'pending', 'screening', 'interview', 'final', 'completed'

CREATE INDEX idx_analyses_pipeline
ON analyses(user_email, pipeline_stage);

신규 테이블

-- candidate_notes (후보자 메모)
CREATE TABLE candidate_notes (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  analysis_id UUID REFERENCES analyses(id) ON DELETE CASCADE,
  user_email TEXT NOT NULL,
  note TEXT NOT NULL CHECK (length(note) <= 500),
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_notes_analysis
ON candidate_notes(analysis_id);

-- Row Level Security
ALTER TABLE candidate_notes ENABLE ROW LEVEL SECURITY;
CREATE POLICY "사용자는 본인 메모만 조회"
ON candidate_notes FOR SELECT
USING (auth.jwt() ->> 'email' = user_email);

-- candidate_tags (후보자 태그)
CREATE TABLE candidate_tags (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  analysis_id UUID REFERENCES analyses(id) ON DELETE CASCADE,
  user_email TEXT NOT NULL,
  tag TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW(),
  UNIQUE(analysis_id, tag)
);

CREATE INDEX idx_tags_analysis
ON candidate_tags(analysis_id);
CREATE INDEX idx_tags_user
ON candidate_tags(user_email, tag);

-- Row Level Security
ALTER TABLE candidate_tags ENABLE ROW LEVEL SECURITY;
CREATE POLICY "사용자는 본인 태그만 조회"
ON candidate_tags FOR SELECT
USING (auth.jwt() ->> 'email' = user_email);

-- pipeline_history (단계 이동 히스토리)
CREATE TABLE pipeline_history (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  analysis_id UUID REFERENCES analyses(id) ON DELETE CASCADE,
  user_email TEXT NOT NULL,
  from_stage TEXT,
  to_stage TEXT NOT NULL,
  created_at TIMESTAMP DEFAULT NOW()
);

CREATE INDEX idx_history_analysis
ON pipeline_history(analysis_id);

5. 보안 체크리스트 🔐

⚠️
중요: Security by Design
2026.06.13 localStorage 개인정보 유출 사건 교훈 - 모든 기능 구현 전 보안 설계 필수!

✅ 데이터 저장 시

  • ☑ 사용자별 데이터 분리 (user_email 컬럼)
  • ☑ Supabase RLS (Row Level Security) 정책 적용
  • ☑ 민감정보 암호화 (필요 시)
  • ☑ 다른 사용자 접근 차단

✅ 데이터 조회 시

  • ☑ 세션 인증 체크 (auth())
  • ☑ 본인 데이터만 조회 (WHERE user_email = ?)
  • ☑ URL 파라미터 조작 방지
  • ☑ API 엔드포인트 인증/인가

✅ API 설계 시

  • ☑ 모든 엔드포인트에 auth() 체크
  • ☑ Input validation (Zod 스키마)
  • ☑ SQL Injection 방지 (Prepared Statement)
  • ☑ XSS 방지 (입력값 sanitize)
  • ☑ Rate limiting (과도한 요청 차단)

보안 코드 패턴 예시

// ✅ 올바른 API 패턴
export async function POST(request: Request) {
  // 1. 인증 체크
  const session = await auth()
  if (!session?.user?.email) {
    return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
  }
  const userEmail = session.user.email

  // 2. Input validation
  const body = await request.json()
  const { analysisId, stage } = body

  // 3. 본인 데이터인지 확인
  const { data: analysis } = await supabase
    .from('analyses')
    .select('user_email')
    .eq('id', analysisId)
    .single()

  if (analysis?.user_email !== userEmail) {
    return NextResponse.json({ error: 'Forbidden' }, { status: 403 })
  }

  // 4. 작업 수행
  const { error } = await supabase
    .from('analyses')
    .update({ pipeline_stage: stage })
    .eq('id', analysisId)
    .eq('user_email', userEmail)  // 이중 체크!

  if (error) {
    return NextResponse.json({ error: error.message }, { status: 500 })
  }

  return NextResponse.json({ success: true })
}
